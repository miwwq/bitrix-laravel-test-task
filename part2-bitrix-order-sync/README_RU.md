# Часть 2. Bitrix: отправка заказа в Laravel API

## Что делает код

Обработчик подписывается на событие D7 `sale:OnSaleOrderSaved`. После сохранения заказа он собирает основные данные заказа, свойства и товары корзины, затем отправляет JSON во внешний Laravel API.

Если Laravel API недоступен, заказ всё равно сохраняется:

- HTTP-запрос обернут в `try/catch`;
- таймаут ограничен тремя секундами;
- ошибка пишется в лог через `AddMessage2Log`;
- исключение не пробрасывается пользователю.

## Как сделать фоновую отправку

Более надежный вариант:

1. В обработчике `OnSaleOrderSaved` не делать HTTP-запрос.
2. Сохранить задачу в отдельную таблицу, например `b_order_export_queue`: `ORDER_ID`, `STATUS`, `RETRY_COUNT`, `NEXT_ATTEMPT_AT`, `LAST_ERROR`.
3. Агент или cron-команда выбирает несколько задач со статусом `new/error`.
4. Отправляет их в Laravel API.
5. При успехе ставит `done`.
6. При ошибке увеличивает `RETRY_COUNT`, пишет `LAST_ERROR` и переносит `NEXT_ATTEMPT_AT`.

Для высокой нагрузки лучше cron, например раз в минуту, потому что агенты Битрикса завязаны на хиты сайта, если не вынесены на cron. Так пользователь не ждет внешний API, а временная недоступность Laravel не мешает оформлению заказа.
